# AI Harness Engineer 视频策划包

**日期**: 2026-03-28  
**阶段**: 研究后策划  
**定位**: 面向开发者与 AI 工程从业者的竖屏知识短视频  
**建议时长**: 95-125 秒  
**建议基调**: 强判断、轻危机、重信息密度

## 1. 一句话主张

`AI Harness Engineer` 不是一个已经彻底标准化的新岗位，但它对应的工作正在快速成形: 工程师的核心价值，正在从“自己写代码”转向“让 AI 代理稳定、可控、可验证地交付结果”。

## 2. 受众

### 核心受众

- 已经在用 Cursor、Claude Code、Codex、Windsurf 的开发者
- 对 agents、MCP、evals、workflow automation 感兴趣的人
- 想判断自己会不会被 AI 改写工作方式的技术观众

### 非核心但可兼容受众

- 泛科技内容受众
- 想转 AI 工程方向的后端或全栈开发者

## 3. 这条视频要解决的问题

让观众在 2 分钟内明白三件事：

1. `AI Harness Engineer` 不是空词，它背后已经有真实的一线信号  
2. 真正重要的不是 prompt，而是代理的外层系统  
3. 如果你已经在做 sandbox、tools、evals、approvals、observability，你其实已经在接近这个角色

## 4. 推荐标题

### 主标题

- Prompt之后是Harness

### 备选标题

- Harness新工程角色
- 最值钱的不是写代码
- AI时代新工程角色

## 5. 开场 Hook

### 推荐 Hook 版本

你以为 AI 时代最值钱的是提示词。  
但 2026 年真正开始冒头的，可能是另一个角色。  
不是 Prompt Engineer。  
而是 `AI Harness Engineer`。

### 备选 Hook 版本

大模型越来越强。

而是代理外面那层系统。

## 6. 核心叙事

这条视频不讲“一个突然爆红的新职位”，而讲：

**软件工程的价值重心正在上移。**

从：

- 自己写代码

转向：

- 设计环境
- 管理权限
- 组织工具
- 建立评测
- 观察行为
- 验证结果

## 7. 7 段结构

### Scene 1 - Hook

**标题**: Prompt 不够了  
**目标**: 先打破观众的旧认知  
**讲述重点**:

- 大家以为 Prompt Engineer 是终局
- 但代理真正进入生产后，问题不在 prompt
- 新瓶颈在模型外层系统

**建议画面**:

- `Prompt Engineer` 字样被压缩到角落
- 中央浮出 `Harness Engineer`
- 屏幕上快速闪过 `tools / memory / evals / approvals / traces`

### Scene 2 - 痛点

**标题**: Demo 能跑，生产跑不动  
**目标**: 让观众产生真实共鸣  
**讲述重点**:

- 很多人做 agent demo，只接一个 tool 就能演示
- 但一进真实环境，就会撞到权限、上下文漂移、回放困难、失败不可复现
- 企业买的不是会演示的 agent，而是可靠代理

**建议画面**:

- 左边 `demo` 只有 `prompt + tool`
- 右边 `production` 展开成 `prompt + tools + memory + evals + approvals + observability + runtime`

### Scene 3 - 定义

**标题**: 什么是 Harness  
**目标**: 给出简单、可传播的定义  
**讲述重点**:

- Harness 不是单个框架
- 它是让代理稳定工作的整套运行外壳
- 更适合的定义是: 把 AI 代理变成可上线、可审计、可复用能力的人

**建议画面**:

- 中心圆写 `Agent`
- 外圈依次点亮 `Context / Tools / Memory / Sandbox / Approval / Evals / Traces`

### Scene 4 - 证据 1

**标题**: OpenAI 已经下场  
**目标**: 给观众最硬的一手信号  
**讲述重点**:

- OpenAI 2026 年明确写出 `Harness engineering`
- 他们用 Codex 做了真实产品，并把工程重心描述成设计环境和反馈闭环
- Codex harness / App Server 也证明这已经是产品和平台层能力

**建议画面**:

- OpenAI 风格信息卡
- 放大几个关键词: `Humans steer. Agents execute.` `feedback loops` `app server`

### Scene 5 - 证据 2

**标题**: 能力已经被拆开  
**目标**: 证明这不是抽象空谈  
**讲述重点**:

- LangChain 已经把 harness 能力拆成规划、文件系统、子代理、上下文管理、执行、人类审批、skills、memory
- MCP 也在强调边界和用户同意
- 这说明行业在往“系统工程”收敛

**建议画面**:

- 模块栈图
- `Planning` `Subagents` `Execute` `Human-in-loop` `Memory` 逐个点亮

### Scene 6 - 机会

**标题**: 谁会先吃到红利  
**目标**: 把结论拉回观众自身  
**讲述重点**:

- 真正会涨价值的，不只是写 prompt 的人
- 而是能让代理稳定交付结果的人
- 如果你已经在做 workflow、tooling、sandbox、evals、guardrails、observability，你就在往这个方向走

**建议画面**:

- 职业迁移路径图
- `Prompt` → `Context` → `Harness`

### Scene 7 - CTA

**标题**: 现在就该补的能力  
**目标**: 让观众知道下一步做什么  
**讲述重点**:

- 少盯一点花哨 demo
- 多补运行环境、评测、权限、可见性
- 未来真正稀缺的，是把代理变成可靠系统的人

**建议金句**:

- 以后最值钱的，不只是会写代码的人  
- 而是能让 AI 稳定交付的人

## 10. 旁白表达原则

- 不要把它说成已被所有公司采用的正式岗位
- 不要贬低 Prompt Engineering 的历史价值
- 要强调“岗位名未统一，但工作已真实存在”
- 要多用“正在形成”“开始浮出水面”“对应职责已经出现”这类稳健表达

## 11. 可直接进入脚本阶段的简版旁白骨架

你以为 AI 时代最值钱的是提示词。  
但 2026 年真正开始冒头的，可能是另一个角色。  
AI Harness Engineer。  

为什么会出现这个词？  
因为大家慢慢发现，模型越来越强，可真实上线最难的部分，已经不是 prompt。  
而是模型外面的整套运行系统。  

一个 agent demo 能跑，不代表它能进生产。  
权限、上下文、评测、回放、观测、审批，只要少一个，代理就可能失控。  

OpenAI 今年已经明确在讲 Harness Engineering。  
LangChain 也把 harness 的能力拆得很清楚。  
这些信号都在说明一件事。  

以后更值钱的，不只是会写代码的人。  
而是能把 AI 代理变成可靠系统的人。  

如果你已经在做 tools、sandbox、evals、guardrails、workflow、observability。  
那你其实已经在往这个角色走了。
